PlanRun · аудит существующего

Что есть сейчас, как это работает, что мешает тренерам. База для редизайна.

TL;DR

PlanRun — приложение для бегунов и тренеров на React + Capacitor (Android-APK). Ядро — план тренировок: AI-generated или ручной, разбит на недели → дни → planDays (один день = массив тренировок: бег + ОФП + СБУ). Бегун видит дашборд с виджетами, календарь, статистику и чат с AI. Тренер видит список учеников с метриками compliance + переключается в режим просмотра конкретного ученика через URL ?athlete=slug. Главная проблема UX тренера — работа с одним учеником за раз через тот же UI, что и у бегуна; масс-операции (bulk-edit, групповые тренировки) отсутствуют. Дизайн-система — Strava-orange + Apple Liquid Glass.

01 Архитектура и навигация

AppLayout + BottomNav
components/AppLayout.jsx, components/common/BottomNav.jsx

Все авторизованные экраны живут под одним лейаутом: TopHeader + BottomNav + Outlet. Один и тот же layout для обеих ролей — меняются только лейблы вкладок.

02 Дэшборд бегуна USER

Dashboard.jsx — конструктор виджетов
components/Dashboard/Dashboard.jsx (784 строки) + 7 виджетов

Главный экран ученика. Самое необычное — пользователь сам собирает себе дашборд: drag-and-drop виджеты в одну или две колонки. Layout сохраняется в localStorage.

9 виджетов (можно включать/выключать, переупорядочивать)

  • goal_countdown — обратный отсчёт до главной гонки + фаза макроцикла (база/build/peak/taper) GoalCountdownWidget.jsx
  • today_workout — карточка сегодняшней тренировки (WorkoutCard)
  • next_workout — следующая запланированная тренировка
  • trend_compare — тренд этой недели vs прошлой
  • personal_records — личные рекорды по дистанциям
  • race_prediction — VDOT-прогноз + темповые зоны (Riegel) RacePredictionWidget.jsx
  • training_load — TRIMP / ATL / CTL / TSB / ACWR (TrainingPeaks-style) TrainingLoadWidget.jsx
  • calendar — DashboardWeekStrip (компактная неделя)
  • stats — DashboardStatsWidget (суммы за неделю)

Скрытое: DailyBriefingHero

Над виджетами рендерится hero с проактивным брифингом от AI-тренера. Cron генерит сообщение утром (event_key = coach.proactive_daily_briefing), кликом открывается чат. Если брифинга нет — карточка не показывается.

Pull-to-refresh + auto-refresh

На мобильном — pull-to-refresh; в фоне — useWorkoutRefreshStore.startAutoRefresh() опрашивает API каждые 30 сек (на webе) или ловит resume (на native). Это нужно, чтобы Strava-webhook долетел до UI без перезагрузки.

Empty-states

  1. Onboarding не пройден → «Добро пожаловать» + кнопка SpecializationModal
  2. Plan не существует и AI-режим → «Создайте план» + кнопка regenerate
  3. Plan генерится → баннер «Это займёт 3-5 минут»
  4. Plan-error → красный notice + retry
  5. Сегодня нет тренировки → «День отдыха»

Layout-конструктор

Через @dnd-kit. Десктоп: можно собрать 2 виджета в одну строку (merge-zone), мобайл: всегда по одному. dashboardLayout.js хранит layout как [[id], [id1, id2], [id]...].

UX-проблема: 9 виджетов + конструктор — это здорово, но это дизайн под продвинутого пользователя. Новичок не знает, какие виджеты ему нужны и в каком порядке. И главное — тренер этим же экраном пользоваться не может; у него / — это AthletesOverview, совсем другой экран. То есть Dashboard сделан только под бегуна.

03 Экран тренера COACH

AthletesOverviewScreen — главный экран тренера
screens/AthletesOverviewScreen.jsx (598 строк)

Что показывается

  1. Фильтр по группам (chips сверху) — Все / [группы тренера] / Создать группу
  2. Секция «Требуют внимания» — атлеты с одним из триггеров:
    • Нет ни одной отмеченной тренировки
    • ≥7 дней без активности
    • Compliance < 50% от запланированного «к сегодняшнему дню»
  3. Секция «Остальные» — все остальные ученики, сортируемые: по активности / по имени / по compliance

Карточка ученика (AthleteCard)

Аватар · имя · бейдж «Новое» (если есть свежая активность) · дата последней активности · цель (тип забега + целевое время) · дата гонки с countdown · полоска compliance · теги групп · 3 кнопки действий: Календарь / Профиль / Написать.

Группы (GroupsModal)

Тренер может создавать группы (имя + цвет из 8 палитр), управлять участниками через чекбоксы. Используются для фильтрации списка атлетов. Но самих групповых действий через UI нет — нельзя «назначить тренировку группе» в один клик.

Как тренер реально работает

AthletesOverview → click «Календарь» на карточке → /calendar?athlete=ivanov → работает с планом ученика через те же UI-компоненты, что и сам ученик

То есть тренер использует UI бегуна с подменой viewContext. У него нет своего «тренерского» представления плана.

Главная боль тренера (по словам пользователя): «не удобно пользоваться всем функционалом». Что конкретно мешает:
  • Нет bulk-операций: чтобы назначить интервалы 5 атлетам — нужно открыть каждого по очереди.
  • Нет одной точки для реакции: чтобы понять «кто загрузил тренировку», «кто пропустил», «кто задал вопрос» — нужно переходить между списком атлетов, чатом, календарями.
  • Контекст теряется при drill-in: открыл календарь ученика → ушёл из списка → нельзя сравнить с другими.
  • Сравнение атлетов отсутствует: compliance видно только в карточке, нельзя поставить 3-4 человек рядом и увидеть разницу.
  • Календарь тренера = тот же календарь, что у ученика, но с возможностью править. UI оптимизирован под просмотр одной недели одного человека, а не под групповое планирование.

04 Календарь и тренировки

Структура данных плана
components/Calendar/Calendar.jsx, Week.jsx, Day.jsx, WorkoutCard.jsx
plan.weeks_data[] → week.days{ mon, tue, wed, thu, fri, sat, sun } → день = [planDay, planDay...] (массив!)

Один день — это массив planDays. Может быть несколько тренировок (например, утро = СБУ, вечер = темповый бег).

Типы тренировок (TYPE_NAMES в WorkoutCard.jsx)

easy · лёгкий бег tempo · темповый interval · интервалы long · длительный control · контрольный fartlek · фартлек race · соревнование sbu · СБУ other · ОФП rest · отдых walking / hiking / cycling / swimming

Структурированные упражнения

Для ОФП и СБУ описание парсится в массив: {name, sets, reps, weight, duration, distance} → отображается списком с чипами «3×12», «20 кг», etc. utils/structuredExercises.js + parseStructuredExercises()

Status workout: planned / completed / missed / rest

«Completed» проверяется кросс-источникно: buildProgressDataMap сверяет план с реальными workout-логами (Strava/Polar/Coros/manual), result-ами и executedExercises для ОФП. Workout считается выполненным, если категория плана совпала с категорией факта на ту же дату.

WorkoutCard.jsx — сердце UI
471 строка — самый сложный компонент

Эта одна карточка переиспользуется везде: дэшборд, календарь, статистика, профиль. Поэтому у неё мегамного пропсов: workout, status, isToday, compact, planDays, workoutMetrics, results, canEdit, extraActions, maxDescriptionItems...

Что внутри

  • Заголовок: тип тренировки + дата + бейдж «Сегодня»
  • Цветная полоска слева (по типу: бег=оранжевый, ходьба=зелёный, СБУ=violet, ОФП=красный)
  • Список planDays с кнопками «Изменить» и «Удалить» (если canEdit)
  • Описание — либо HTML (для бега), либо структурированный список упражнений (для ОФП/СБУ)
  • Метрики: дистанция + время + темп (из workoutMetrics или results или регулярка по тексту)
  • extraActions слот — куда дашборд кидает кнопки «Отметить выполнение» / «Развернуть»
DayModal.jsx — детальный просмотр дня
602 строки

Открывается по клику на день в календаре. Показывает: план дня + фактически загруженные тренировки + результаты + заметки (могут писать ученик и тренер). Внутри встроен AddTrainingModal для добавления/правки тренировки и WorkoutDetailsModal для анализа конкретной тренировки с графиками.

AddTrainingModal.jsx — конструктор тренировки
1188 строк — самый большой компонент

Два шага: выбор категории (Бег / ОФП / СБУ) → форма. Для бега есть калькулятор: вводишь любые две из трёх метрик (дистанция, время, темп) — третья считается автоматически. Для ОФП/СБУ — конструктор упражнений с sets×reps×weight.

06 Чат и AI

ChatScreen.jsx — двухколоночный чат
1056 строк

3 типа чатов

  • AI-тренер — главный «звёздный» чат. SSE-стрим из api/chat_sse.php. AI может вызывать tools: get_training_day, update_training_day, swap_training_days, log_workout, recalculate_plan, get_stats_summary... Показывает индикаторы «Смотрю план…», «Обновляю тренировку…».
  • От администрации — broadcast от админов
  • Личные диалоги — direct messages (тренер ↔ ученик; пользователь ↔ пользователь). Если ты coach, у тебя тут все ученики.

Quick replies

Подсказки внизу: «Как мне сегодня тренироваться?», «Поменяй темп завтра». chatQuickReplies.js + SUGGESTED_PROMPTS.

UX

Mobile: список чатов → выбор → разговор (один экран). Desktop: список слева, диалог справа. Pin-to-bottom, keyboard-aware (chat-keyboard-open класс). Стриминг с фазами: connecting → streaming → tool:... → done.

08 Дизайн-система · что есть

sports-colors.css + buttons.css + cards.css
src/styles/*

Цвета

  • Primary: Strava Orange #FC4C02 со шкалой 50–900
  • Success #22C55E (выполнено), warning #EAB308 (темп), danger #EF4444 (риск/интервалы)
  • Workout colors — отдельные семантические токены для типов тренировок
  • Workout strips — цветные полоски на карточках для категорий (бег/ходьба/поход/СБУ/ОФП)

Типографика

Montserrat (UI, 300–800) + Jost для цифр и статистики. Спортивная пара, считывается мгновенно.

Поверхности — Apple Liquid Glass

Карточки имеют backdrop-filter: blur(20px) saturate(1.16), многослойные тени с оранжевым tint, radial-gradient overlays. Это даёт «премиальный» вид, но стоит производительности на mid-range Android — в коде есть html.native-app с уменьшенным blur.

Темы

Light (F4F7FB фон с радиальными оранжевыми пятнами) + Dark (Nike-стиль, #0B1015). Полный /design-system экран для админов как living-doc.

Bottom-nav — floating pill

Парящий навбар position: fixed; bottom: 12px с blur + анимированная таблетка под активным табом.

10 Что я предлагаю менять

Главное

Тренер сейчас работает в «гостевом режиме» — переиспользует UI бегуна. Это нужно принципиально разделить: у тренера должны быть свои first-class экраны для:

  1. Реакции в потоке — лента событий «кто что загрузил / пропустил / спросил»
  2. Команды как целого — heatmap compliance всей группы, сравнение N атлетов
  3. Bulk-действий — назначить тренировку 5 атлетам, применить шаблон, написать в группу
  4. Drill-in без потери контекста — детали ученика как панель/overlay, а не как смена страницы

А у бегуна — упростить дашборд: не 9 виджетов на выбор, а 1 фокусный экран «что у меня сегодня» с возможностью свайпнуть к графикам. 9 виджетов хороши для power-users, но обычный бегун теряется.

Что точно сохранить

  • Strava-orange как primary
  • Семантические цвета типов тренировок (это уникальная сила)
  • Jost для цифр
  • AI-чат с tool-calling (это киллер-фича — единственное приложение, где AI может реально править план)
  • VDOT / TRIMP / ATL / CTL — серьёзная аналитика, нельзя выкидывать
  • Многоисточниковая выгрузка (Strava + Polar + Coros + Suunto + Telegram)

Что переделать

  • UI тренера → новые first-class экраны (см. редизайн A/B/C)
  • Liquid Glass → оставить как опцию, но дать «density mode» (тренеру нужна плотность данных)
  • Bottom-nav для тренера → переосмыслить: возможно, «Поток / Команда / Календарь / Чат / Аналитика»
  • Дашборд бегуна → дать «простой» и «продвинутый» режимы
